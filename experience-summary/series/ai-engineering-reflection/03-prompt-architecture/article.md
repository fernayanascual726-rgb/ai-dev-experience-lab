# Prompt：当代开发者的临时架构文档

当 AI 开始参与真实软件工程后，Prompt 不再只是输入框里的几句话，而是在逐渐承担系统约束与上下文传递。

这不是一篇 Prompt 技巧文章。

这里讨论的不是如何写出“万能提示词”，也不是总结某个工具的使用方法，而是一个更工程化的问题：当 AI Coding 进入日常开发流程后，Prompt 正在软件工程体系里扮演什么角色？

我的观察是：Prompt 正在从一次性指令，演变为一种临时、实时、可执行的架构文档。

它承载的不只是“让 AI 做什么”，还包括：系统当前是什么样、哪些边界不能突破、哪些历史约束必须保留、代码应该遵守什么规范、输出需要被怎样验证。

换句话说，Prompt 开始成为 AI Native Engineering 里的上下文接口。

---

## 1. 引言：传统架构文档正在失去同步能力

传统软件工程里，架构文档通常承担几个职责：

- 解释系统为什么这样设计
- 记录模块边界和依赖关系
- 描述关键流程和状态变化
- 保留历史决策和技术取舍
- 帮助新人理解系统上下文

这些职责仍然重要。

但现实问题是，传统文档很难保持同步。

很多团队都有类似经历：

- README 只在项目初始化时认真写过
- Wiki 里的架构图已经落后几个版本
- 设计文档记录了最初方案，但没有记录后续妥协
- 关键约束只存在于老开发者的记忆里
- 代码已经演化，文档却没有跟上

过去，这个问题已经存在。AI Coding 出现后，它被进一步放大。

因为 AI 会显著提高代码和方案的生成速度。系统变化速度开始超过传统文档维护速度。

当一个开发者让 AI 修改代码时，AI 需要的不是三个月前的设计文档，而是当前任务的实时上下文：这个模块现在有哪些约束？哪些接口不能改？哪些历史坑不能踩？哪些规范是团队默认遵守但没有写进代码的？

于是，很多团队真实的“上下文入口”开始发生变化。

它不再只是：

```text
README
设计文档
Wiki
```

而越来越多地变成：

```text
Prompt
```

Prompt 开始接管一部分传统文档没有及时承担的职责。

但这也带来新的问题：如果 Prompt 成为实际工作流里的上下文入口，它就不再只是个人输入习惯，而是一种工程结构。

## 2. Prompt 正在承载越来越多的信息

最早使用 AI Coding 时，Prompt 往往很简单：

```text
帮我写一个用户注册接口。
```

但在真实工程环境里，这样的 Prompt 很快就不够用。

因为 AI 不知道系统中的隐性约束，也不知道历史上为什么形成现在的设计。

于是 Prompt 开始变长，开始包含越来越多工程信息。

### 2.1 业务约束

业务约束是 AI 最容易缺失、也最容易造成隐性问题的部分。

例如，一个订单系统里的业务规则可能包括：

- 只有 `created` 状态的订单可以取消
- 支付成功回调必须幂等
- 企业客户可以走账期，普通客户必须即时支付
- 风控未通过时不能创建发货任务
- 退款金额不能超过实付金额

这些规则不一定集中写在一个地方。

有些在代码里，有些在测试里，有些在产品文档里，有些只存在于团队共识里。

当开发者让 AI 生成订单取消逻辑时，如果 Prompt 只是：

```text
帮我实现订单取消接口。
```

AI 可能会生成一个局部可用的实现，但遗漏状态约束、幂等处理或风控条件。

因此开发者开始在 Prompt 中补充业务规则：

```text
实现订单取消逻辑。

业务约束：
- 只有 created 状态可以取消
- paid 状态需要走退款流程，不能直接取消
- cancellation_reason 必须记录
- 重复取消请求需要保持幂等
- 风控锁定的订单不能取消
```

这段 Prompt 已经不是简单指令，而是在向 AI 传递业务模型。

### 2.2 架构约束

AI 生成代码时，经常会选择“最短路径”。

但软件工程里的正确路径，往往不是最短路径，而是符合系统架构的路径。

例如团队可能有这样的约束：

- Controller 不能直接访问数据库
- 所有写操作必须经过 Service 层
- Repository 只负责数据读写，不包含业务判断
- 外部 API 调用必须经过 Gateway
- 不能绕过已有的领域对象直接拼装状态

这些约束如果没有进入 Prompt，AI 很可能生成类似代码：

```python
def cancel_order(request):
    order = db.orders.find_one({"id": request.order_id})
    order["status"] = "cancelled"
    db.orders.save(order)
```

这段代码局部看很直接，但它绕过了 Service、Repository、状态机和审计记录。

于是 Prompt 开始承担“架构约束描述”：

```text
请实现订单取消功能。

架构约束：
- 不允许 Controller 直接访问数据库
- 必须通过 OrderService 完成状态流转
- 状态变更必须使用 OrderStateMachine
- 数据持久化只能通过 OrderRepository
- 需要复用已有 AuditLogService 记录操作日志
```

这已经非常接近一段架构说明。

它不是为了让 AI “更听话”，而是为了让 AI 的输出不破坏系统结构。

### 2.3 编码规范

很多团队也开始把编码规范直接写进 Prompt。

例如：

- 命名规则
- 错误处理方式
- 日志规范
- 测试要求
- 返回值结构
- 空值处理策略

一个真实的 Prompt 可能是这样：

```text
编码规范：
- 函数命名使用动词 + 领域对象
- 不允许 except Exception 后静默返回
- 业务异常使用 DomainError
- 基础设施异常交给上层 middleware
- 关键状态变更必须记录 info 日志
- 为新增分支补充单元测试
```

这不是 Prompt 技巧，而是团队规范在 AI Workflow 中的再表达。

过去这些规则可能写在贡献指南、代码 Review 经验或团队约定里。

现在，如果它们不进入 Prompt，AI 就很难稳定遵守。

### 2.4 历史上下文

AI Coding 最大的问题之一，是 AI 不知道“系统为什么会变成现在这样”。

它看到的是当前代码片段，但看不到背后的历史原因。

例如：

- 为什么这里没有使用某个更通用的抽象
- 为什么这个接口不能改名
- 为什么这个字段虽然看起来冗余但不能删除
- 为什么某段逻辑不能合并到公共方法里
- 为什么这个模块不能直接依赖另一个模块

这些历史上下文往往决定了系统边界。

开发者不得不把这些内容写进 Prompt：

```text
历史背景：
- OrderStatus.confirmed 不能改名，外部 BI 报表依赖该字段值
- LegacyPaymentGateway 仍在被三个老渠道使用，不能直接删除
- 这里没有合并到 CommonOrderService，是因为企业订单和渠道订单的事务边界不同
- 不要把确认逻辑抽到 util，必须保留在领域服务中
```

这类内容非常像架构决策记录。

但它出现的位置不是 ADR 文档，而是 Prompt。

这就是 Prompt 角色变化的关键：它开始承载系统历史。

## 3. 一个真实工程案例

一个团队开始大规模使用 AI Coding 来改造内部运营系统。

这个系统已经运行多年，技术栈不复杂，但业务规则很多。里面有订单、库存、优惠券、支付、审批、通知等模块。

最初，开发者给 AI 的 Prompt 很短：

```text
帮我新增一个批量审批订单的接口。
```

AI 很快生成了 Controller、Service、Repository 调用和测试草稿。

第一版看起来效率很高。

但进入 Review 后，问题开始出现。

### AI 破坏了已有抽象

系统中原本有一个 `ApprovalPolicy`，所有审批条件都应该通过它判断。

但 AI 直接在 Service 里写了条件：

```python
if order.amount < 5000 and order.status == "pending":
    order.approve()
```

这段代码局部正确，但绕过了已有策略对象。

后续审批规则变化时，这里很容易被遗漏。

### AI 绕过了基础设施

系统中所有批量任务都应该通过 `JobScheduler` 执行，以便统一重试、限流和监控。

但 AI 生成了一个同步循环：

```python
for order_id in order_ids:
    approval_service.approve(order_id)
```

短期看能跑。

但如果订单数量很大，会导致接口超时，也无法接入已有监控。

### AI 重复实现了逻辑

系统里已经有一个 `BatchApprovalService`，但因为当前 Prompt 没有提到，AI 新增了 `BulkOrderApprovalService`。

两个 Service 名字不同，逻辑相似，边界模糊。

这种重复不会立刻出错，却会让系统概念越来越混乱。

### AI 忽略了历史约定

过去团队曾经踩过一个坑：审批成功后不能立即发送通知，因为审批结果还需要经过异步风控确认。

这个约定没有写在当前代码片段里。

AI 生成了：

```python
order.approve()
notification.send_approval_success(order.user_id)
```

这看起来很自然，但它违反了历史约定。

Review 时，老开发者指出这会导致用户收到过早通知。

### Prompt 开始变成系统上下文入口

经历几次类似问题后，团队开始改 Prompt。

新的 Prompt 不再只是需求描述，而是包含：

```text
任务：新增批量审批订单能力。

模块边界：
- Controller 只负责参数解析和响应，不包含审批逻辑
- 审批规则必须通过 ApprovalPolicy
- 批量任务必须通过 JobScheduler，不允许同步循环处理
- 数据访问必须经过 OrderRepository

历史背景：
- 审批成功后不能立即通知用户，需要等待异步风控确认
- BatchApprovalService 已经存在，优先复用，不要新增平行 Service

编码规范：
- 业务异常使用 DomainError
- 不允许吞掉异常
- 需要补充单元测试，覆盖空列表、部分失败和重复请求
```

结果很明显：AI 输出质量提升了。

它更少绕过已有抽象，更容易复用已有 Service，也更能生成符合团队约定的测试。

但团队也意识到一个更深的问题：真正决定代码质量的，已经不只是模型能力，而是 Prompt 中承载了多少系统上下文。

Prompt 成了系统知识的临时集装箱。

它越完整，AI 输出越接近团队期望。

它越缺失，AI 越容易局部正确、全局偏离。

## 4. Prompt 正在演变成“临时架构文档”

Prompt 和传统架构文档不同。

它不是一个正式文档，也不一定会被长期保存。但在任务执行期间，它确实承担了架构约束传递的职责。

这就是 Prompt 正在变成“临时架构文档”的原因。

### 4.1 实时性

传统文档经常是：

```text
写完就过期。
```

不是因为团队不重视文档，而是因为系统演化太快，文档维护成本太高。

Prompt 则不同。

Prompt 会随着当前任务实时更新。

当开发者发现 AI 忽略了幂等，就补充幂等要求。

当 AI 绕过 Repository，就补充架构边界。

当 AI 误解状态流，就补充状态转移规则。

它不一定完整，但它非常贴近当前任务。

### 4.2 可执行性

传统文档只是描述。

Prompt 会直接影响系统生成结果。

这是一件很大的变化。

一段架构文档写着“Service 不应直接访问数据库”，开发者可能遵守，也可能忘记。

但一段 Prompt 写着：

```text
不允许 Service 直接访问数据库，所有数据读写必须通过 Repository。
```

AI 的输出会立刻受到影响。

Prompt 不是单纯记录知识，而是在驱动生成行为。

这让 Prompt 接近一种可执行的工程约束。

### 4.3 上下文绑定

Prompt 不只是知识，它是当前任务上下文。

它天然绑定：

- 当前模块
- 当前需求
- 当前状态
- 当前约束
- 当前 Review 目标

同一个系统，不同任务需要的 Prompt 可能不同。

Debug Prompt 需要日志、错误路径和复现条件。

Code Review Prompt 需要架构边界、风险点和修改意图。

测试生成 Prompt 需要业务边界、异常路径和验收条件。

Prompt 的价值不只是“通用模板”，而是把通用规则和当前任务上下文组合在一起。

### 4.4 临时性

Prompt 很多时候不会被永久保存。

这也是它的风险。

但不能因为它临时，就低估它在工程流程里的作用。

在任务执行期间，Prompt 已经承担了架构约束传递的职责。

它决定 AI 如何理解系统，也影响最终代码是否符合团队预期。

如果一个 Prompt 里包含了关键业务规则、架构边界和历史约束，却没有被沉淀，那么这部分知识就会在 Session 结束后消失。

这会形成新的知识断层。

## 5. Prompt 资产化：团队开始沉淀新的工程资产

随着 AI Coding 深入，很多团队会开始沉淀 Prompt。

这不是因为 Prompt 本身神秘，而是因为 Prompt 本质上正在变成“系统行为约束”。

团队可能会逐渐积累：

- Prompt 模板
- Workflow Prompt
- Code Review Prompt
- 架构 Prompt
- Debug Prompt
- Agent Prompt

例如，一个 Code Review Prompt 可能会要求 AI 检查：

```text
请从以下维度 Review 本次改动：

1. 是否破坏模块边界
2. 是否重复实现已有逻辑
3. 是否遗漏异常处理、权限、幂等和日志
4. 是否引入新的隐性复杂度
5. 是否需要补充测试或监控
```

这已经不是“提示词技巧”。

它是在把团队的 Review 经验编码成可复用的协作入口。

一个架构 Prompt 可能会描述：

```text
当前系统分层：
- API 层只负责请求解析
- Application Service 负责编排流程
- Domain Service 负责业务规则
- Repository 负责数据访问

请不要跨层调用，不要在 API 层写业务判断。
```

这也不是教程，而是架构约束的再表达。

AI 时代，团队真正的工程资产，不再只有代码。

而是：

```text
代码 + Prompt + Workflow + Context
```

代码描述系统当前状态。

Prompt 描述 AI 应该如何理解和改变这个状态。

Workflow 描述人和 AI 如何协作。

Context 描述为什么系统不能随意改变。

这些内容一起，才构成 AI Native Engineering 的工程资产。

---

## 6. Prompt 的风险：新的上下文债务

Prompt 资产化并不意味着 Prompt 越多越好。

Prompt 也会失控。

它可能出现：

- 冗余
- 过期
- 冲突
- 无限堆叠
- 风格不统一
- 历史规则残留

例如，一个团队不断把遇到的问题追加到同一个 Prompt 里：

```text
不要直接访问数据库。
不要吞异常。
不要新增 util。
不要改接口。
不要改状态名。
不要生成超过三个文件。
不要使用旧的 PaymentClient。
不要忘记通知。
不要立即通知。
不要重复通知。
```

这样的 Prompt 会越来越像一份失控的临时规范。

它记录了很多真实问题，但缺少结构和治理。

一段时间后，团队可能没人知道：

- 哪些规则仍然有效
- 哪些规则只适用于某个模块
- 哪些规则来自历史事故
- 哪些规则已经被新的架构替代
- 哪些规则彼此冲突

这就是 Prompt 作为上下文债务的表现。

它和《上下文正在变成新的技术债》里的问题是同一条线：当上下文没有被管理，它就会从帮助系统理解，变成阻碍系统演化。

Prompt 本身也需要维护。

需要有人判断：哪些内容应该进入长期文档，哪些应该进入模板，哪些只适合当前任务，哪些应该被删除。

否则，Prompt 会从“临时架构文档”变成“临时垃圾场”。

---

## 7. 总结：软件工程正在变成上下文工程

Prompt 很重要，但这不是结论。

更重要的变化是：AI 正在把软件工程从代码工程，逐渐推向上下文工程。

过去，开发者主要管理代码：模块、接口、依赖、测试、发布。

现在，开发者还要管理 AI 理解系统所需要的上下文：业务规则、架构边界、历史原因、Review 经验、工作流状态。

Prompt 不只是人与 AI 的对话。

它正在成为 AI Native Engineering 时代的临时架构载体。

它实时、可执行、绑定任务，也充满风险。

如果团队能够把 Prompt 中的有效约束沉淀为模板、文档、测试和工作流，它会成为新的工程资产。

如果团队只是不断堆叠 Prompt，却不治理上下文，它也会成为新的技术债。

这就是 Prompt Architecture 真正值得讨论的地方。

不是如何写一句更聪明的话，而是如何让 AI 在真实工程系统里，持续理解正确的边界。
