# AI Workflow 的核心已经不是生成，而是状态管理

当 AI 开始进入真实工作流后，真正复杂的问题，已经不再是“生成答案”，而是：如何管理持续演化的系统状态。

过去一段时间，行业讨论 AI 的方式，很大程度上仍然围绕生成能力展开。

模型能不能写代码，能不能理解长上下文，能不能推理复杂问题，能不能一次生成更完整的方案。

这些问题当然重要。

但当 AI 从单次问答进入真实工程流程后，系统的复杂度会发生变化。AI 不再只是接收一个 Prompt，然后返回一个 Output。它开始参与多步骤任务，跨越多个系统，调用工具，写入状态，读取历史，恢复中断任务，并在不同 Agent 之间传递上下文。

这时，问题的核心就开始从“AI 会不会生成”转向“AI 系统如何长期稳定运行”。

我的观察是：AI Workflow 的核心，正在从生成能力转向状态管理能力。

---

## 1. 引言：行业仍然过度关注“生成”

现在大多数 AI 讨论仍然集中在几个方向：

- 模型能力
- Context Window
- 推理能力
- Prompt 技巧
- 输出质量

这些讨论背后有一个隐含前提：只要模型足够强，Prompt 足够清楚，输出质量足够高，AI 系统就会变得可靠。

但真实工程环境里的情况更复杂。

很多 AI 系统失败，并不是因为模型“不够聪明”。

它们失败，是因为系统状态开始失控。

在 Demo 阶段，每次运行都是全新上下文。

开发者打开一个工具，输入一段需求，AI 生成代码、总结文档或给出方案。即使结果不好，也可以重新开始。上一轮失败不会太多影响下一轮，因为整个过程更像一次性的生成任务。

但生产环境不是这样。

真实 AI Workflow 会不断积累历史状态。

它可能包含：

- 半完成任务
- 过期上下文
- 冲突记忆
- Retry 状态
- 中断 Workflow
- 错误恢复链路
- 已经执行过但没有正确记录的外部操作
- 多个 Agent 对同一任务形成的不同判断

这些状态不会因为一次对话结束就自然消失。

一个客服自动化系统可能记住了用户昨天的投诉，但没有记录今天已经人工处理过。

一个 AI Coding Agent 可能知道自己改过某个文件，但不知道测试失败后开发者已经手动回滚了一部分修改。

一个财务审批 Workflow 可能完成了外部 API 调用，但内部任务状态仍然停留在“执行中”。

这些问题看起来不像传统意义上的生成错误。

模型没有写错一句话，也没有明显理解错任务。真正的问题是：系统不知道自己处在什么状态。

这就是 AI Workflow 进入真实工程环境后最容易被低估的部分。

AI 生成能力越强，系统越容易进入长链路；Workflow 越长，状态越多；状态越多，失败模式越接近传统复杂系统，而不是简单的 Prompt 优化问题。

---

## 2. AI 系统为什么正在“状态化”

传统 ChatBot 的结构很简单：

```text
Prompt → Output
```

这是一种接近无状态的交互。

用户输入问题，模型生成回答。下一次输入可以带上历史，也可以不带。即使有多轮对话，状态主要仍然存在于上下文窗口中。

但 Agent 和 Workflow 出现后，结构开始变化。

系统开始拥有：

- Memory
- History
- Task State
- Execution State
- Recovery State
- Context Graph
- Tool Call Result
- Human Feedback
- Agent Coordination State

这意味着 AI 系统不再只是“生成文本的接口”。

它开始变成一个持续运行的状态系统。

一旦系统拥有“持续状态”，问题的本质就变了。

它不再只是：

```text
如何让模型这一次回答正确？
```

而是：

```text
如何让系统在多次运行、多次中断、多次恢复、多次上下文变化之后，仍然保持正确？
```

这就是状态化带来的根本变化。

### 2.1 状态会改变系统行为

状态不是被动记录。

状态会改变系统行为。

一个 AI Agent 如果记住了“上次这个用户喜欢保守方案”，它下次可能会自动选择更保守的实现。

一个 AI Coding 系统如果摘要了“这个模块不能修改接口”，后续任务可能会避免触碰接口，即使新的需求已经允许接口演进。

一个 Workflow 如果记录某个步骤已经完成，恢复执行时就会跳过它；如果这个记录不准确，系统就可能跳过必要步骤，或者重复执行已经完成的动作。

状态一旦进入系统决策路径，它就不只是信息，而是行为约束。

这会带来很多新的失败模式：

- 历史错误会污染后续决策
- 旧上下文可能覆盖新状态
- Retry 可能导致重复执行
- 长期 Memory 会逐渐漂移
- 错误摘要会被当成事实继续传播
- 人工修正没有进入状态系统，导致 AI 继续沿用旧判断

例如，一个 AI Review Agent 在早期记录了：

```text
当前项目不使用数据库事务。
```

这个判断当时可能是对的，因为它只看到了一个只读模块。

但如果这条记忆进入长期 Memory，后续它在 Review 支付写入逻辑时，可能继续默认“项目不使用事务”，从而遗漏事务边界问题。

表面看，这是 AI 推理失败。

但本质上，是状态来源错误、适用范围不清、过期信息没有被淘汰。

很多所谓的“AI 推理失败”，本质上其实是状态管理失败。

### 2.2 状态让 AI 系统开始接近“分布式系统”

当 AI Workflow 继续变复杂，它会越来越像分布式系统。

这不是一个比喻，而是工程结构上的相似。

现在的 AI Workflow 已经开始出现：

- 状态同步
- 上下文传播
- 幂等问题
- Retry 问题
- 回滚问题
- Workflow 恢复
- 并发 Agent
- 状态一致性
- 外部工具调用的副作用管理

一个多 Agent 系统里，规划 Agent 生成任务拆分，执行 Agent 调用工具，Review Agent 检查结果，记忆模块保存摘要，Orchestration Layer 决定是否重试。

这里的每一步都可能产生状态。

如果规划 Agent 的任务状态没有同步给执行 Agent，执行 Agent 可能重复执行已经被取消的任务。

如果 Review Agent 看到的是旧版本文件，它可能基于过期上下文给出错误判断。

如果 Memory 保存的是失败前的摘要，恢复后系统可能认为某个假设仍然成立。

这已经不是单模型推理问题。

它更像一个由多个组件、多个状态源、多个执行阶段组成的系统一致性问题。

AI 工程问题，正在越来越像“分布式系统问题”。

过去我们在分布式系统里讨论幂等、重试、补偿、事务边界、消息重复、状态一致性。现在，这些问题正在 AI Workflow 中以新的形式重新出现。

只是这一次，状态里不只有数据库字段和消息队列事件，还包括自然语言上下文、模型摘要、工具调用记录、Agent 判断和人类反馈。

这让问题更难。

因为自然语言状态不像数据库记录那样稳定。它可能模糊、压缩、遗漏、带有解释性，也可能在多轮传递中逐渐变形。

## 3. 一个真实工程案例

有一个团队尝试构建 AI 自动化工作流，用来处理企业客户的财务申请。

早期流程很简单。

运营人员提交一段申请说明，AI 读取材料，生成审核摘要，判断是否缺少信息，并给出下一步建议。

整个流程基本是：

```text
Prompt → 生成 → 成功
```

效果很好。

AI 能快速总结材料，发现缺失字段，也能把复杂说明整理成结构化审批意见。团队很快把它扩展成一个更完整的 Workflow：

1. 读取申请材料
2. 识别申请类型
3. 检查企业资质
4. 调用内部风控接口
5. 调用银行预校验 API
6. 生成审批建议
7. 通知运营人员
8. 写入审批系统

Demo 阶段看起来很顺利。

但进入真实业务后，问题开始出现。

### 中途任务失败

某次 Workflow 执行到银行预校验时，外部 API 超时。

系统捕获了错误，并触发 Retry。

问题是，AI Agent 的执行日志里记录了“银行预校验已发起”，但 Workflow State 仍然停留在“等待调用银行 API”。

于是恢复任务时，系统不知道这次调用到底有没有真正发出。

如果只看自然语言日志，像是已经调用过。

如果只看结构化状态，像是还没调用。

两个状态源发生了冲突。

### 重试导致重复动作

另一次 Workflow 中，系统已经成功调用银行 API，银行侧也返回了预校验通过。

但在写入内部审批系统时，数据库连接短暂失败。

Orchestrator 判断任务失败，于是重新执行整个步骤。

银行 API 本身做了幂等，所以没有产生资金风险。

但通知模块没有幂等。

Retry 后，系统重复发送了通知：

```text
您的企业申请已通过银行预校验。
```

用户收到两次相同通知，运营人员也看到两条记录。

这不是模型不会生成。

AI 的审批摘要是正确的，调用参数也是正确的，通知文案也没有问题。

真正的问题是：Workflow 没有清楚地区分“外部 API 已完成”“内部写入失败”“通知是否已经发送”。

系统不知道什么已经执行，什么失败了，什么可以重试，什么必须跳过或补偿。

### Memory 污染后续决策

更隐蔽的问题出现在 Memory。

为了让 AI 更了解客户，团队让系统保存每次审批摘要。

某个客户早期因为材料不完整，被记录为：

```text
该客户历史申请材料质量较差，需要重点核验。
```

这条记忆在当时有用。

但几个月后，这个客户已经完成了资料整改，并被人工标记为“资料可信度恢复正常”。

问题是，人工标记进入了审批系统，却没有同步进入 AI Memory。

后续 AI 在处理该客户的新申请时，仍然检索到旧摘要，并倾向于给出更保守的审批建议。

从表面看，AI 像是在“过度谨慎”。

但根因不是模型性格保守，而是长期 Memory 失真。

### Agent 决策漂移

随着 Workflow 变复杂，团队又增加了多个 Agent：

- Intake Agent 负责材料解析
- Risk Agent 负责风险判断
- Policy Agent 负责规则匹配
- Notification Agent 负责通知生成
- Review Agent 负责最终检查

一开始，每个 Agent 的职责很清楚。

但随着 Prompt 不断追加规则，Agent 边界开始变模糊。

Risk Agent 会给出审批建议，Policy Agent 也会给出审批建议，Review Agent 又会基于两者生成一个新的判断。

当三者结论不一致时，系统没有明确的状态合并规则。

于是有时以 Risk Agent 为准，有时以 Policy Agent 为准，有时最终摘要又覆盖了前面的结构化判断。

这类问题很难通过“换更强模型”解决。

因为问题已经不再是“AI 会不会生成”。

而是系统如何知道：

- 什么已经执行
- 什么失败了
- 什么可以重试
- 什么必须回滚
- 哪个状态源是权威的
- 哪个 Agent 的判断可以覆盖另一个 Agent
- 哪些自然语言摘要只能作为参考，不能作为执行依据

当 AI Workflow 进入真实业务，这些问题会比生成质量本身更决定系统可靠性。

## 4. AI Workflow 的真正核心：状态管理

未来 AI Workflow 的核心能力，不是生成能力，而是状态治理能力。

生成能力仍然重要。

但它更像底层能力。真正决定系统能否进入生产环境的，是系统能否稳定管理上下文、任务、执行、恢复、观察和长期记忆。

### 4.1 上下文管理

上下文管理的核心问题不是：

```text
能塞进多少上下文？
```

而是：

```text
什么应该进入上下文？
什么应该被遗忘？
```

Context Window 增大，并不等于上下文问题消失。

更大的窗口只是允许系统携带更多信息。

但更多信息不一定带来更好判断。它也可能带来噪音、冲突、过期约束和错误关联。

一个 AI Coding Agent 如果把过去所有 Review 评论都塞进上下文，可能会变得过度保守。

一个客服 Agent 如果把用户历史所有投诉都作为当前任务背景，可能会把已经解决的问题重新拿出来影响判断。

一个审批 Agent 如果检索到旧政策，又同时检索到新政策，但没有明确优先级，就可能生成看似合理但规则混杂的结论。

真正困难的是上下文选择。

好的 Context Engineering 不是无限堆叠信息，而是判断：

- 当前任务需要哪些上下文
- 哪些上下文只是背景，不应影响决策
- 哪些上下文已经过期
- 哪些上下文需要结构化表达
- 哪些上下文必须来自权威系统
- 哪些上下文只能作为模型参考，不能作为执行依据

上下文管理，本质上是状态边界管理。

### 4.2 状态恢复

真实 Workflow 一定会中断。

外部 API 会超时，工具调用会失败，人工审批会延迟，Agent 可能生成不确定结果，系统也可能在某个步骤之后崩溃。

因此，未来 AI Workflow 会越来越像可恢复工作流系统。

Workflow 中断后，系统需要知道：

- 已经完成哪些步骤
- 哪些步骤产生了外部副作用
- 哪些步骤可以安全重试
- 哪些步骤需要补偿或回滚
- 哪些状态需要人工确认
- 哪些上下文需要重新拉取而不是沿用旧缓存

这里最关键的是：AI Workflow 不能只保存自然语言过程摘要。

自然语言摘要适合帮助人理解，但不适合作为唯一执行状态。

例如：

```text
已完成银行预校验，等待后续处理。
```

这句话对人有用，但对系统不够。

系统还需要知道：

```text
bank_precheck.status = succeeded
bank_precheck.request_id = xxx
bank_precheck.idempotency_key = yyy
notification.sent = false
approval_record.persisted = false
```

只有这样，恢复执行时系统才能避免重复调用、重复通知或跳过必要写入。

AI Workflow 的状态恢复，不只是“继续刚才的对话”。

它是工程系统里的恢复语义。

### 4.3 状态可观测性

传统软件系统里，工程师可以依赖日志、Stack Trace、指标、链路追踪和 Replay。

但 AI Workflow 的失败更难观察。

很多失败是概率性的、上下文相关的、不可稳定复现的。

同一个 Prompt，在不同上下文、不同 Memory、不同工具返回、不同中间摘要下，可能走向完全不同的结果。

因此，AI Workflow 的可观测性不能只记录最终输出。

它还需要记录：

- 当时检索到了哪些上下文
- 哪些 Memory 被注入了 Prompt
- 哪些工具被调用
- 工具返回了什么
- Agent 做了哪些中间判断
- 哪些状态被写入
- 哪些步骤被跳过
- 哪些人类反馈改变了执行路径

否则，当系统出错时，团队只能看到一个结果，却不知道这个结果是如何形成的。

这会让 Debug 变得非常困难。

例如，一个 AI Agent 错误批准了一笔申请。

如果只有最终审批文本，团队很难判断：

- 是模型理解错了政策
- 是检索到了旧政策
- 是 Risk Agent 输出被错误覆盖
- 是人工修正没有进入上下文
- 是恢复任务时跳过了某个校验步骤

这些问题都可能导致同一个结果，但修复方式完全不同。

AI Workflow 的可观测性，未来会成为工程系统的核心能力之一。

### 4.4 长期 Memory 漂移

长期 Memory 经常被描述为 AI 系统变聪明的关键。

但长期 Memory 并不天然可靠。

它也会漂移。

这种漂移可能来自：

- 旧记忆污染新任务
- 错误摘要逐渐累积
- Retrieval 偏移
- 历史状态失真
- 临时规则被保存成长期规则
- 局部经验被错误推广为全局约束

例如，一个 AI Coding Agent 在某次任务里被告知：

```text
这个模块不要新增抽象。
```

这句话可能只适用于当时的小修复。

但如果它被保存成长期偏好，后续在真正需要抽象的重构任务中，Agent 可能会继续避免抽象，导致代码复制增加。

Memory 的价值在于帮助系统延续经验。

Memory 的风险也在这里：它可能让系统延续错误经验。

因此，长期 Memory 需要治理。

它需要来源、时间、适用范围、置信度、过期机制和人工修正入口。

否则，Memory 会从能力变成负担。

## 5. 辩证思考：AI 越强，状态问题越严重

很多人以为：

```text
模型越强，问题越少。
```

这在单次生成任务里有一定道理。

更强的模型通常能更好地理解需求，更少犯低级错误，也能处理更复杂的上下文。

但在真实 AI Workflow 里，现实可能相反：

```text
AI 越强，系统状态越复杂。
```

因为模型越强，人们越愿意把更长、更复杂、更关键的任务交给它。

于是：

- Workflow 更长
- Agent 更多
- 自动化更深
- 上下文持续时间更久
- 外部工具调用更多
- 人机协作边界更复杂
- 系统需要恢复和追踪的状态更多

弱 AI 只能完成短任务，所以状态问题不明显。

强 AI 能进入长链路，所以状态问题会被放大。

这和软件系统的发展很像。

一个简单脚本不需要复杂架构。

但当它开始承载生产业务、被多人调用、依赖外部系统、需要失败恢复和审计追踪时，它就不再只是脚本，而是系统。

AI Workflow 也一样。

当 AI 只是生成一段文本，问题主要是输出质量。

当 AI 开始持续参与业务流程，问题就变成系统治理。

最终，系统复杂度增长速度，开始超过人的认知速度。

这句话很关键。

AI 提升了执行速度，也提升了系统变化速度。

过去，一个复杂 Workflow 可能需要团队手动设计、评审、实现和测试，因此它的增长速度相对可控。

现在，AI 可以快速生成新的 Agent、新的流程、新的 Prompt、新的工具调用和新的异常分支。

每一处新增都可能合理。

但它们组合起来，会形成一个人很难完整理解的状态空间。

团队以为自己只是增加了几个自动化步骤，实际上却增加了许多隐含状态：谁负责决策，哪些上下文有效，失败后如何恢复，重试是否安全，人工修改是否覆盖 AI 判断，旧 Memory 是否仍然适用。

当状态空间扩张速度超过团队理解速度，可靠性就会下降。

这不是 AI 不够强。

这是系统工程没有跟上 AI 能力扩张。

---

## 6. 前沿方向：Context Engineering 正在出现

行业已经开始从 Prompt Engineering 转向 Context Engineering。

这不是说 Prompt 不重要。

而是说，仅仅讨论“怎么写 Prompt”已经不够了。

AI 系统真正需要解决的问题正在变成：

```text
如何管理：
- Context
- Memory
- Workflow
- State
- Agent Coordination
```

这也是为什么越来越多前沿方向开始围绕 AI Orchestration Layer 展开。

例如：

- Multi-Agent Orchestration
- Context Graph
- Persistent Memory
- Durable Workflow
- Workflow Replay
- Agent State
- 状态恢复

这些方向表面上看很技术化，但背后指向的是同一个问题：如何让 AI 系统在长时间、多步骤、多组件协作中保持一致。

Context Graph 试图解决上下文之间的关系问题。

Persistent Memory 试图解决经验延续问题。

Workflow Replay 试图解决失败复盘和状态重建问题。

Durable Workflow 试图解决中断恢复和副作用管理问题。

Multi-Agent Orchestration 试图解决多个 Agent 之间的职责、顺序和协调问题。

这些不是简单的框架能力。

它们代表 AI 工程正在从“调用模型”走向“治理系统”。

Prompt Engineering 关注的是如何把意图表达给模型。

Context Engineering 关注的是如何让系统在正确的时间，把正确的上下文，以正确的结构，交给正确的执行单元，并在执行后正确更新状态。

这比写一句好 Prompt 更难。

也更接近真实工程。

---

## 7. 最终升维：AI 正在把软件工程重新拉回“系统工程”

最后，问题不只是 Workflow 很重要。

更大的变化是：AI 正在让软件工程重新回到“系统工程”。

过去几年，行业有时过度关注局部效率：

- 前端框架
- 生成速度
- API 拼装
- 单点工具能力
- 更快完成一个页面或接口

这些当然有价值。

但 Agent 和 Workflow 出现后，真正重要的东西重新浮现出来：

- 状态
- 边界
- 一致性
- Workflow
- 可恢复性
- 系统治理
- 可观测性
- 长期维护成本

AI 让生成变便宜。

但生成变便宜之后，系统问题不会消失，反而会更快暴露。

因为当代码、文档、流程、Agent 和自动化步骤都可以快速生成时，真正稀缺的能力就变成：判断这些东西如何组成一个稳定系统。

这也是 AI Native Engineering 和传统“工具使用”的区别。

如果 AI 只是一个生成器，那么我们只需要关心输出是否足够好。

但如果 AI 是工作流的一部分，是系统执行路径的一部分，是团队协作和业务决策的一部分，那么我们必须关心它如何保存状态、恢复状态、传播上下文、处理失败、接受人工修正，并在长期运行中保持一致。

AI Native Engineering 的核心，可能从来都不是“生成”。

而是：如何让复杂系统，在持续变化中，依然保持稳定。

这不是一个短期工具问题。

它更像软件工程正在进入下一阶段时，重新遇到的老问题：当系统变复杂之后，工程的核心永远不是让某个局部更快，而是让整体仍然可理解、可恢复、可治理。

AI Workflow 只是把这个问题提前推到了我们面前。
